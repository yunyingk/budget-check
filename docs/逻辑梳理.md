# 预算校验逻辑梳理

## 1. 入口

- 接口: `POST /api/ebot/check`
- 入参: `ticket_id`(单据ID), `callback_url`(回调地址)
- 异步处理，立即返回 `{code: 200, message: "处理中"}`

## 2. 单据字段提取

从合思 API 查单据详情，提取 4 个关键字段:

| 字段 | 来源 | 说明 |
|------|------|------|
| 成本中心 | `form.E_system_costcenter` | 部门/成本中心 ID |
| 费用档案 | `form.details[0].feeTypeForm.u_费用类型档案` 或 `form.u_费用类型档案` | 费用类型档案 ID |
| 项目 | `form.项目` | 项目 ID |
| 费用性质 | `form.u_费用性质` | 映射为下表 |

费用性质映射:

| 合思 ID | 中文 |
|---------|------|
| `ID01LPD78hZRsr` | 业务 |
| `ID01LPDisfN3qv` | 管理 |
| `ID01LPDfjPcnyn` | 生产 |
| 其他 | 未知 |

## 3. 校验规则

### 3.1 成本中心校验 & 费用档案校验（逻辑相同）

在 `budget_cache` 表中查 `dim_code`，**不限制 dim_type**:
- 找到 → 通过
- 没找到 → 调合思 API 向上追溯 parentId，最多 5 层
  - 某层父级在库里 → 通过
  - 追到底没找到 → 不通过

### 3.2 项目校验（仅生产费用需要）

在 `budget_cache` 表中查 `dim_code`，**不限制 dim_type**:
- 找到 → 通过
- 没找到 → 调合思 API 查该项目的 name:
  - name 含 "公摊成本" → **豁免，直接通过**
  - 否则 → 不通过

### 3.3 最终判定

```
if 费用性质 == "业务" or "管理":
    if 成本中心OK and 费用档案OK:
        → 通过
    else:
        → 拒绝（列出缺的项）

elif 费用性质 == "生产":
    if 项目是"公摊成本":
        if 成本中心OK and 费用档案OK:
            → 通过(公摊豁免)
        else:
            → 拒绝（列出缺的项）
    else:
        if 成本中心OK and 费用档案OK and 项目OK:
            → 通过
        else:
            → 拒绝（列出缺的项）

elif 费用性质 == "未知":
    → 拒绝（未配置的性质ID）

else:
    → 拒绝（暂不支持的性质）
```

## 4. 回调

校验完成后 POST 回调:
```json
{
    "ticket_id": "xxx",
    "status": "PASS | REJECT",
    "message": "通过 | 成本中心不在预算内、项目不在预算内 | ..."
}
```
