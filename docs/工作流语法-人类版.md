# 工作流语法（人类版）

## 概述

每个预算包（target）是一个独立的工作流，包含一系列 **steps**（步骤），按顺序执行。

工作流的核心思想：
1. **初始数据集**：单据本身（1 条记录）
2. **逐步处理**：每一步可以拆分数据、过滤数据、或执行校验
3. **最终校验**：剩余的每条记录都需要通过预算包匹配

---

## 配置结构

```json
{
  "version": 1,
  "targets": [
    {
      "id": "预算包ID",
      "name": "预算包名称",
      "steps": [
        { "description": "步骤说明", ... },
        { "description": "步骤说明", ... }
      ]
    }
  ]
}
```

- `version`：固定为 1
- `targets`：预算包数组，每个 target 独立执行
- `steps`：步骤数组，按顺序执行

---

## Step 类型

### 1. 数据变换动作（改变数据集）

#### `split_detail` - 按费用明细拆分

将单据的 `details` 数组拆分为多条记录。

**输入**：1 条记录，包含 `details` 数组
**输出**：N 条记录，每条对应一个 detail

```json
{
  "description": "按费用明细拆分",
  "action": "split_detail"
}
```

**数据流示例**：
```
输入：
{
  "金额": 10000,
  "details": [
    { "费用类型": "差旅费", "金额": 6000 },
    { "费用类型": "办公费", "金额": 4000 }
  ]
}

输出：
[
  { "金额": 10000, "费用类型": "差旅费", "金额": 6000 },
  { "金额": 10000, "费用类型": "办公费", "金额": 4000 }
]
```

**特殊处理**：
- `details` 中的 `feeTypeForm` 字段会展开到顶层
- 删除 `details` 字段，避免递归

---

#### `split_apportion` - 按分摊比例拆分

将当前记录按 `apportions` 数组拆分为多条记录。

**输入**：N 条记录，每条包含 `apportions` 数组
**输出**：M 条记录，每条对应一个 apportion

```json
{
  "description": "启动分摊",
  "action": "split_apportion"
}
```

**数据流示例**：
```
输入：
{
  "费用类型": "差旅费",
  "金额": 6000,
  "apportions": [
    { "apportionForm": { "比例": 0.6 } },
    { "apportionForm": { "比例": 0.4 } }
  ]
}

输出：
[
  { "费用类型": "差旅费", "金额": 6000, "比例": 0.6 },
  { "费用类型": "差旅费", "金额": 6000, "比例": 0.4 }
]
```

**注意**：
- 必须在 `split_detail` 之后执行（apportions 嵌套在 details 里）
- `apportions` 中的 `apportionForm` 字段会展开到顶层
- 删除 `apportions` 字段

---

### 2. 条件判断（when/then）

根据条件决定如何处理当前记录。

```json
{
  "description": "条件说明",
  "when": "表达式",
  "then": "pass | refuse | commit"
}
```

#### `when` 表达式

使用 [expr-lang](https://github.com/expr-lang/expr) 语法，可用字段包括：
- 单据顶层字段：`u_费用性质`、`E_system_costcenter`、`项目` 等
- detail 字段：拆分后会合并到顶层
- apportion 字段：拆分后会合并到顶层

**常用表达式**：
```javascript
// 等于
u_费用性质 == 'ID01LPDfjPcnyn'

// 不等于
u_费用性质 != 'ID01LPDfjPcnyn'

// 包含
项目 in ['ID01LZNNxip807', 'ID01LZNNxip808']

// 不包含
项目 not in ['ID01LZNNxip807']

// 逻辑组合
u_费用性质 == 'ID01LPDfjPcnyn' && 项目 != 'ID01LZNNxip807'
```

#### `then` 动作

| 动作 | 含义 |
|------|------|
| `pass` | 该记录通过，从数据集中移除，不再执行后续 steps |
| `refuse` | 拒绝整个单据，可配置 `reason` 说明原因 |
| `commit` | 该记录提交，保留在数据集中，但后续非-split steps 跳过它 |

#### 示例

```json
{
  "description": "费用性质非生产时免项目预算校验",
  "when": "u_费用性质 != 'ID01LPDfjPcnyn'",
  "then": "pass"
}
```

含义：如果费用性质不是"生产"，则该记录通过，不需要校验项目预算。

```json
{
  "description": "指定项目免预算校验",
  "when": "项目 == 'ID01LZNNxip807'",
  "then": "pass"
}
```

含义：如果项目是"公摊成本"，则该记录通过，不需要校验。

```json
{
  "description": "缺少项目时拒绝",
  "when": "项目 == ''",
  "then": "refuse",
  "reason": "生产费用缺少项目"
}
```

含义：如果项目为空，拒绝整个单据。

### 3. 预算包匹配（match_info_to_budget）

将当前记录的字段与预算树逐层匹配。

```json
{
  "description": "匹配预算包",
  "action": "match_info_to_budget"
}
```

**匹配逻辑**：
1. 从预算树根节点开始
2. 取当前层的维度字段名（`dimensionId`）
3. 从记录中取该字段的值
4. 在当前层的节点中查找
   - PROJECT 类型：向上找祖先（最多 5 层）
   - 其他类型：精确匹配
5. 匹配成功 → 进入下一层
6. 匹配失败 → 拒绝，提示"xxx 不在预算包内"

**示例**：
```
预算树：
├── 成本中心层（dimId: E_system_costcenter）
│   ├── 北京研发中心
│   │   ├── 项目A
│   │   └── 项目B
│   └── 上海销售部

记录字段：
{
  "E_system_costcenter": "北京研发中心ID",
  "项目": "项目AID"
}

匹配过程：
1. 成本中心层：北京研发中心ID → 找到 → 进入下一层
2. 项目层：项目AID → 找到 → 匹配成功
```

## 完整示例

### 成本中心预算（简单三步）

```json
{
  "id": "ID01TsPQJFK1RR",
  "name": "成本中心预算",
  "steps": [
    { "description": "按费用明细拆分", "action": "split_detail" },
    { "description": "启动分摊", "action": "split_apportion" },
    { "description": "匹配成本中心预算包", "action": "match_info_to_budget" }
  ]
}
```

### 项目预算（带条件判断）

```json
{
  "id": "ID01T5kHipEY7J",
  "name": "项目预算",
  "steps": [
    {
      "description": "费用性质非生产时免项目预算校验",
      "when": "u_费用性质 != 'ID01LPDfjPcnyn'",
      "then": "pass"
    },
    { "description": "按费用明细拆分", "action": "split_detail" },
    {
      "description": "指定项目免预算校验",
      "when": "项目 == 'ID01LZNNxip807'",
      "then": "pass"
    },
    { "description": "启动分摊", "action": "split_apportion" },
    { "description": "匹配项目预算包", "action": "match_info_to_budget" }
  ]
}
```

执行流程：
1. 检查费用性质，非生产费用直接通过
2. 拆分明细
3. 检查项目，公摊成本直接通过
4. 拆分分摊
5. 匹配预算包

---

## 执行规则

1. **顺序执行**：steps 按数组顺序执行
2. **数据传递**：每一步的输出是下一步的输入
3. **独立执行**：每个 target 独立执行，互不影响
4. **任一拒绝则拒绝**：所有 target 都通过才算通过

### 数据集变化

```
初始：[单据]
  ↓ split_detail
[明细1, 明细2, ...]
  ↓ when/then (pass)
[未通过的明细]
  ↓ split_apportion
[分摊1, 分摊2, ...]
  ↓ when/then (refuse)
拒绝 或 继续
  ↓ match_info_to_budget
匹配成功 或 拒绝
```

## 常见问题

**Q: split_detail 和 split_apportion 的顺序？**

必须先 `split_detail`，后 `split_apportion`。因为 apportions 嵌套在 details 里。

**Q: when 表达式可以用哪些字段？**

所有单据字段都可以用，包括：
- 顶层字段：`u_费用性质`、`E_system_costcenter`、`项目` 等
- detail 字段：拆分后会合并到顶层
- apportion 字段：拆分后会合并到顶层

**Q: pass 和 commit 的区别？**

- `pass`：从数据集中移除，后续步骤看不到它
- `commit`：保留在数据集中，但后续非-split steps 跳过它

**Q: 多个 target 怎么执行？**

每个 target 独立执行，互不影响。所有 target 都通过才算通过，任一拒绝则拒绝。

**Q: 可以没有 steps 吗？**

不可以。每个 target 必须有至少一个 step。
